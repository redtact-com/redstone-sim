# 13. スコープ/ポジショニング方針 (#62)

redstone-sim が「何であり、何でないか」を定める方針文書。個別コンポーネントの採否根拠 (10) や
更新抑制のスコープ判断 (12 §5) の上位に位置し、今後のスコープ判断はまず本書に照らす。
ユーザ方針の原文は issue #62 に記録 (2026-07-03)。

---

## 1. 位置づけ — ゲームではなく「回路の学習・開発ツール」

redstone-sim は **Web 上で動くレッドストーン回路シミュレータ**であり、提供する価値は次の 3 つ:

1. **回路共有サイト (redtact) との連携** — 手元に PC / ゲーム本体が無くても、共有された回路の
   動作をブラウザで確認できる
2. **解説記事への埋め込み** — 記事内で回路を実際に動かして見せられる (tick 送り・トレース付き)
3. **学習環境 + 回路勢の開発環境** — 実機 fixture で検証済みの vanilla 忠実な挙動 (更新順・
   within-tick 順まで) を、freeze/step/トレースという実機より観察しやすい形で提供する

ゲームプレイ (サバイバル・建築・戦闘・収集) は提供しない。**「遊ぶ」ものではなく「調べる・
学ぶ・設計する」もの**という位置づけを UI/README/対外説明で一貫させる。

## 2. 誤解リスクと「エンティティ境界」原則

高機能化するほど「マイクラのパクリゲーム」と誤解されるリスクが上がる。これを避けるため、
機能の採否は次の 1 原則で判定する:

> **エンティティ境界原則 — ワールドにエンティティを存在させない。**
> アイテムはコンテナ内の数値としてのみ存在し、ワールド空間に出現しない。モブ・プレイヤー・
> 投射物・アイテムエンティティは実装しない。エンティティが本質の挙動は「手動トリガ + 折衷
> モデル」(感圧板・ターゲットの前例) か「非実装」で扱う。

この原則の帰結:

- **一部の回路が動作しないのは許容する** (例: エンティティ検知系・投射物連携・トロッコ回路)。
  非対応は隠さず README / 10 のスコープ宣言で明示する
- 音は鳴らさない (発音イベントの可視化のみ。C5 の前例)
- テクスチャ/モデルは描画のための参照であり、ゲーム的な演出 (パーティクル・BGM・実績等) は
  追加しない

## 3. 機能マトリクス

| 区分 | 機能 | 採否 | 形態 / 根拠 |
|---|---|---|---|
| 回路素子 | ダスト・トーチ・リピーター・コンパレーター・ピストン・オブザーバー等 | **実装済み** | 本体。実機 fixture で検証 (02/10) |
| エンティティ由来の入力 | 感圧板・ターゲット | **実装済み (折衷)** | 手動トリガ + 持続 gt (10 C2-C4)。エンティティ非実装のまま入力を提供 |
| 音 | 音符ブロック | **実装済み (折衷)** | 発音イベントの可視化のみ。音声再生なし (10 C5) |
| ブロック破壊 | **ピストンによる破壊 (PUSH_DESTROY)** | **スコープ入り (新規)** | ダスト等を押すと**アイテム化せず消滅**させる。ドロップ (アイテムエンティティ) を生成しないため境界原則と両立し、vanilla の回路挙動 (押し先のダストが壊れて伸びる) に近づく |
| アイテム物流 | **ホッパー・ドロッパー間のアイテム移動** | **スコープ入り (新規)** | コンテナ内の**数値の移動**として実装 (8gt クールダウン等の回路的挙動が目的)。ワールドへのドロップ・吸い取り (アイテムエンティティ) は対象外 |
| アイテム物流 | ドロッパー/ディスペンサーの射出 (ワールドへの発射) | 非実装 | アイテムエンティティ生成のため境界外。前方がコンテナの場合の**挿入**のみ対象 |
| エンティティ | モブ・プレイヤー・投射物・アイテムドロップ・トロッコ | **非実装 (恒久)** | 境界原則の本体 |
| ワールド | チャンク境界・ディメンション・光量・時刻 | 非実装 | 単一領域 sim (12 §4 と整合) |
| ゲームプレイ | サバイバル要素・建築モード的装飾ブロック群 | 非実装 | 位置づけ (§1)。装飾ブロックは回路に必要な導体 (石等) のみ |

## 4. 新規スコープ入り機能の実装方針 (概要)

### 4.1 ピストンによる破壊 (PUSH_DESTROY)

- 現状: sim は wire/トーチ等を「移動不可」として押し出し自体をブロックする (isMovable 外 =
  障害物)。vanilla は PUSH_DESTROY のブロックを**破壊して**押し出す (回路トポロジーが変わる)
- 方針: collectPushList で PUSH_DESTROY 相当 (wire / torch / lever / button / 感圧板 / dot 等の
  非固体) を「破壊対象」として収集し、移動実行時にアイテム化なしで消滅させる。破壊による
  トポロジー変化は #51 の張り替え機構がそのまま追随する
- 検証: 実機 fixture (押し先にダスト/トーチのある回路)
- **実装済み (#64)**。確定事項 [確定: 26.2 PistonStructureResolver]:
  - 破壊はチェーン終端の 1 ブロック (toDestroy)。そこで連鎖が止まる
  - push limit 12 は toPush のみカウント (破壊対象は数えない — size>=12 チェックが add 前)
  - retract に破壊は無い (DESTROY 分岐は extending 時のみ)。sticky は PUSH_DESTROY を引かず置き去り
  - sim の PushReaction 対応: DESTROY = wire / torch / wall_torch / lever / button 2 種 / 感圧板 4 種 /
    repeater / comparator (いずれも vanilla DESTROY)。
    MOVE = solid / lamp / redstone_block / target / note_block / 非伸長 piston。
    障害物 (押し不可) = extended piston / piston_head / moving_piston / container /
    **observer (vanilla は MOVE だが、移動時のパルス発火の意味論が別途必要なため
    当面障害物のまま — 必要になったら別 issue で可動化)**

### 4.2 ホッパー・ドロッパー物流 (数値モデル)

- 対象: ホッパーの吸い出し/送り込み (コンテナ間)、ドロッパーの前方コンテナへの挿入、
  ロック (NC 受電で停止)、コンパレーター連動 (充填率変化)
- モデル: コンテナは既存 ContainerState (signal 直接指定) を拡張し「アイテム数」を持つ。
  移動は 1 個/8gt (ホッパー) 等の vanilla 値をデコンパイルで確定して数値を動かす。
  スタック種別・スロット配置は持たない (充填率→信号の式は 02 §6 で確定済み)
- 検証: 実機 fixture (ホッパー時計・ドロッパー連鎖等の代表回路)

### 4.3 redtact (回路共有サイト) 連携 — 構想

- 共有された回路 (NBT) を URL で受け取り、閲覧専用/操作可能モードで埋め込み表示する
- 解説記事への iframe/コンポーネント埋め込み。tick 送り・トレース表示を記事側から制御
- 本体リポジトリ (redtact) 側の作業を含むため、要件が固まった時点で両リポに issue を切る

## 5. 運用

- 新機能の提案・要望は本書 §2 の境界原則 → §3 のマトリクスの順に照らして採否を決め、
  結果をマトリクスへ追記する
- 「非実装」の項目は README のスコープ宣言 (10 §10 → README 反映時) にユーザ向けの言葉で
  記載し、代替手段 (手動トリガ・充填率直接指定等) を案内する
