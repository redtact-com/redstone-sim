# 10. 残コンポーネントのスコープ判断たたき台

04 §1 G11 で「未実装コンポーネント」として列挙されながら、I1〜I10 (04 §3) のどの実装 issue にも
割当のないコンポーネント群について、「実装する / スコープ外」の**判断たたき台**をまとめる。
最終決定はユーザが行うため、本書は各コンポーネントに **推奨案 + 根拠 (出典付き) + ユーザ判断ポイント**
を提示する形式に徹し、決定はしない。

- 確度ラベル: **[確定]** = 02 のデコンパイル裏取り or 複数源一致 / **[要検証]** = wiki 等の単一二次源。
- 対象バージョン: **典拠 1.21.1 + 26.x 併読** (02 冒頭 / CONTRIBUTING.md / 04 §4.1)。
- 前提基盤の状況 (04 §3): tile tick 意味論 = **I3 (#11) 済**、weak/strong 電力 = **I2 (#10) 済**、
  更新 3 種 (NC/PP/CU) + 方向順 = **I6 (#14) 未**、ブロックイベントキュー + ピストン + QC = **I7 (#15) 未**、
  コンパレーター完全化 + コンテナ読み = **I5 (#13) 未 (進行中)**。
- 割当 issue の「案」は 04 §3 の I1〜I10 命名を延長した **仮 ID (I11〜I13)**。実 issue 化は親が行う。

## 1. C1: redstone block (レッドストーンブロック)

> **実装済み (I11 / #35)**: BlockType `redstone_block` を追加。全 6 方向に weak 15、固体は強充電しない。
> デコンパイル裏取り (1.21.1 `PoweredBlock`) 済み: `getSignal`=15 / `getDirectSignal` 非 override (=0) /
> `isRedstoneConductor(never)` = 非導体 (ダストの上下斜め接続を切らない)。ダスト 4 面接続・viewer
> (`minecraft:redstone_block`)・nbtIO・editor パレット対応。実機 fixture `redstone-block-static` で
> 静的通電を検証。詳細は 02 §6 redstone block。


- **回路勢重要度: 中**。切れない定数動力源。テクニカル勢では「フライングマシン / スライムブロック建築で
  ピストンに押される可動動力源」として必須級だが、**その用途はピストン (I7) が前提**。ピストン抜きの
  静的回路では「常時 ON のレバー / 灯った torch」と等価の定数源にすぎず、代替が効く。コンパレーター側面入力の
  15 源としても使う (02 §6 comparator の side = レッドストーンブロック 15)。
- **実装コスト: 極小**。依存基盤は **I2 (#10) 済** の weak/strong 電力モデルのみ。挙動は
  「6 方向すべてに弱 15 を出す・強充電はしない (`getDirectSignal`=0)」= 定数の弱動力源。新 BlockType 追加 +
  電源クエリに 1 分岐。tile tick も更新順依存もない。
- **仕様確定状況**: 02 に専用節はないが挙動は単純。wiki [要検証]: 「direct neighbors に**弱** power 15 を供給。
  導体を強充電せず・コンパレーター側面入力に使える」= 弱動力源 15 (強充電なし) と一致。02 §5.2 の
  「弱動力源化した導体は別ダストに伝えない」ルールにそのまま乗る。デコンパイル裏取りは軽微 (RedStoneBlock は
  `isSignalSource=true` / `getSignal`=15 / `getDirectSignal`=0 程度、I11 着手時に確認)。
- **推奨: 実装**。定数の弱 15 源として最小実装。**割当案 = 新規 I11「定数動力源 (redstone block)」**。
  ただし I5 (#13) が既にコンパレーター側面入力で「レッドストーンブロック=15」を扱うため、**I5 に相乗り**して
  BlockType を一緒に足す選択もコスト的に妥当。
- **ユーザ判断ポイント**:
  1. 単独 I11 にするか、I5 (#13) に相乗りさせるか (相乗りの方が総コスト小・ただし I5 のスコープが膨らむ)。
  2. ピストン (I7) 未実装の間は「押せない定数源」に留まる点を許容するか (可動動力源としての本領は I7 後)。

## 2. C2: target block (ターゲットブロック)

> **実装済み (I11 / #35、折衷モデル)**: 当初「スコープ外」推奨だったが、ユーザ判断で「手動トリガ +
> 持続 gt + 全方向 weak」の折衷モデルとして実装した。BlockType `target { outputPower: 0-15 }`。
> activateBlock で命中を模し outputPower=15 (中心命中相当) を出し、**矢の 20 gt 持続** 後に tile tick
> (priority 0) で消灯。デコンパイル裏取り (1.21.1 `TargetBlock`) 済み: 持続 = 矢 20gt (`ACTIVATION_TICKS_ARROWS`) /
> 他 8gt / `tick` で POWER=0 / `getSignal`=POWER (全方向 weak, `getDirectSignal`=0 で強充電なし) /
> `onPlace` が POWER>0 & pending tick 無しを 0 化。ダスト 4 面接続・viewer (`minecraft:target[power=N]`)・
> nbtIO・editor パレット対応。**発火の実機 fixture は不可**: 発火に投射物エンティティが必須で、setblock /
> scarpet の直接 blockstate 変更は onPlace が即 0 に戻す (実機で確認)。消灯系列は手書き単体テスト
> (`power-blocks.test.ts`) で検証。詳細は 02 §6 target。
>
> **追補 (#35 実機バグ報告 2)**: target は導体でもある (isRedstoneConductor=true) ため、ダストが指す
> target は solid と同じ規則で充電され、隣接 lamp / 直上 torch / 背面 comparator に伝導する。
> 他のダストへは伝わらない (shouldSignal。「強給電」の俗説はコード上の根拠なしで棄却)。
> 導体伝導は投射物なしで検証できるため**実機 fixture あり** (`target-conduct`)。詳細は 02 §6 target。


- **回路勢重要度: 低**。信号源としての発火は**投射物 (矢/雪玉/トライデント等) の命中**が条件で、
  ミニゲーム / 的当て用途が主。純粋な回路論理では決定的な信号源にならない。導体として
  「ダストが指して結線でき、被充電もする」性質はコンパクト配線で使われる (wiki) が、その役割は solid で代替可能。
- **実装コスト: 実質不可 (発火機構)**。命中強度は「中心からの距離で 1〜15」・持続は**矢/トライデント 20gt /
  他の投射物 8gt** [要検証: wiki]。いずれも**エンティティ (投射物) シミュレーション**を要し、本 sim には
  エンティティ系が無く I1〜I10 にも計画がない。導体としてのみなら solid 同等で極小だが、値が薄い。
- **仕様確定状況**: 02 に記載なし。強度式・持続 gt とも wiki [要検証] のみ。デコンパイル未着手。
- **推奨: スコープ外**。投射物検知は本 sim の対象系 (静的レッドストーン論理) の外。README に宣言 (§10 草案)。
- **ユーザ判断ポイント**:
  1. 「導体 (solid) としてのみ」置けるようにするか (ダスト結線先を作るためだけ)。実利は薄い。
  2. 将来エンティティ層を入れる計画があるなら再検討トリガーとして残すか。

## 3. C3: 感圧板 wooden / stone

- **回路勢重要度: 低〜中**。ドア・農場・トラップの**入力装置**として頻出だが、機能的には
  「エンティティが乗ると ON」の入力にすぎず、論理素子ではない。sim 上では lever/button と同じ
  「手動トグル入力」で完全に代替できる (回路の下流挙動は入力源の種別に依存しない)。
- **判定対象の差** [要検証: wiki]: 木 = アイテム含む全エンティティで発火 / 石 = Mob・プレイヤーのみ (アイテム不発)。
  出力は ON/OFF の 15。**エンティティ退去後の消灯遅延は 20gt** [要検証: wiki、要デコンパイル裏取り]。
- **実装コスト**: 自動発火は**エンティティ層が前提で不可**。手動入力扱いなら **I4 (#12, ボタン) 済**の
  入力機構に 1 種足すだけで小。消灯遅延 20gt は tile tick (I3 済) に乗る。
- **仕様確定状況**: 02 に記載なし。判定差・遅延とも wiki [要検証]。
- **推奨: 自動発火はスコープ外**。ただし **手動トグル入力としてなら任意で実装可** (lever と重複するため必須ではない)。
- **ユーザ判断ポイント**:
  1. 「乗る/退く」を UI クリックで模す手動入力板を出すか (lever/button で足りるなら不要)。
  2. 出すなら木/石の判定差 (アイテム可否) は手動 sim では意味を持たない点をどう扱うか (見た目だけの別種にするか)。

## 4. C4: 感圧板 weighted (light=金 / heavy=鉄)

- **回路勢重要度: 低**。乗っているエンティティ数に比例したアナログ出力 (アイテムエレベータ検知・
  個数計測) が用途だが、これも**エンティティ計数**が本質。
- **アナログ式** [要検証: wiki、版により定数が変動]: light と heavy で最大重みが異なり
  `出力 = ceil(min(count, maxWeight) * 15 / maxWeight)` 系。**エンティティ数を入力にできない sim では再現不能**
  (手動で「出力レベル」を直接与えるならコンパレーター側の定数源と変わらず、装置としての意味を失う)。
- **実装コスト**: 発火はエンティティ層前提で不可。
- **推奨: スコープ外**。アナログ値が要るならコンパレーター + コンテナ充填 (C6/I5) で代替可能。
- **ユーザ判断ポイント**: **なし (エンティティ計数は再現不能)**。README 宣言のみ。

## 5. C5: note block (音符ブロック)

- **回路勢重要度: 低 (回路論理への影響ほぼ皆無)**。note block は**レッドストーン信号を出力しない**
  (電源ではない) [確定: wiki + 常識]。主機能は音声再生で、論理シミュレータの出力に寄与しない。
  被覆条件 (直上に空気 or 頭 が必要で発音) も音声都合。
- **回路に関わる残り 2 点**: (a) note block は**導体**で、直接充電されると隣を活性化しうる (solid 同等)。
  (b) Java では use/被電/音程変化が**ブロック更新を出しオブザーバーで検知可能** (BUD ネタ)。
  (a) は solid で代替でき、(b) はオブザーバー (I8) 前提のニッチ。
- **実装コスト**: 音声 = スコープ外。BEC (ブロックイベント) 参加は 02 §1.2/§3 [確定] で **I7 (#15) 前提**だが、
  出力が無いので論理再現の価値が立たない。
- **仕様確定状況**: 「回路出力なし・runBlockEvents 参加」は 02 §1.2 (line 39) / §3 (line 113) [確定]。
  楽器/音程/発音は wiki [要検証] だが sim 論理に無関係。
- **推奨: スコープ外 (音声機能)**。置けても論理には効かない。README に宣言。
- **ユーザ判断ポイント**:
  1. 見た目のために「導体 (solid) として」置けるようにするか。
  2. 将来オブザーバー (I8) を入れたとき「note の use/音程変化を PP 発生源にする」ネタ (Java 限定) を拾うか
     — 拾うなら I8 の別スコープ。現時点は不要。

## 6. C6 / C6': コンテナ (chest / hopper / dropper / dispenser / barrel / furnace ほか)

- **回路勢重要度: 高 (充填率読み) / 中 (物流)**。コンパレーターでの**充填率読み取り**は計測回路の中核で重要度高。
  一方、ホッパーのアイテム転送計時 (8gt) やドロッパー/ディスペンサーの発射といった**アイテム物流**は
  自動仕分け・アイテムクロック等で使うが、本 sim の論理範囲からは外縁。
- **C6 充填率読み = I5 担当 (割当済)**。02 §6 comparator の充填率式
  `lerpDiscrete(f,0,15) = floor(f*14) + (f>0?1:0)` [確定: I1 で確定] を、I5 (#13) が
  「コンテナ充填率読み」として実装予定。**本書での新規判断は不要 (I5 #13 に帰属)**。
  ただし sim にアイテム物流が無い以上、I5 のコンテナは**ユーザが充填率 (0〜満杯) を手で与える静的入力**として
  実装されるのが現実的 (実アイテム移動は伴わない)。
- **C6' アイテム物流 = スコープ外 (推奨)**。ホッパー転送・アイテムエンティティ・インベントリ移動は
  **アイテム/インベントリ層**を要し、I1〜I10 に計画なし。ドロッパー/ディスペンサーの **QC は I7 (#15)** が扱う
  (02 §5.3 [確定] の 3 クラス) が、**発射でアイテムを出す挙動自体は物流でスコープ外**。
- **仕様確定状況**: 充填率式 = 02 §6 [確定]。ホッパー計時等の物流仕様は 02 未整理・スコープ外前提で未調査。
- **推奨**: C6 (充填率読み) = **I5 (#13) 担当**。C6' (物流) = **スコープ外**。
- **ユーザ判断ポイント**:
  1. コンテナを「手動で充填率を与える計測用ダミー」に留めてよいか (実アイテム移動を諦めるか)。
     → 計測回路の検証には充填率の直接指定で十分なはず。
  2. ドロッパー/ディスペンサーは I7 で **QC 源 + BEC** としてだけ扱い、アイテム発射はスコープ外で確定するか。

## 7. C7: トーチ burnout (焼き切れ)

- **回路勢重要度: 中**。「RS トーチを速いクロックに入れると焼き切れる」上限はテクニカル勢の常識で、
  トーチクロックの周期下限を規定する。**未実装だと sim 上で本来作れない高速トーチクロックが動いてしまい**、
  実機と乖離する。多くの実用回路は burnout を避ける設計なので日常回路への影響は限定的だが、
  クロック / パルサー系の精度に効く。
- **実装コスト: 小〜中**。依存は **I3 (#11, tile tick) 済**。トーチ単位のトグル履歴 (ワールド単位の
  WeakHashMap 相当) を持ち、tick 時に 60gt より古い記録を捨て、8 件到達で焼き切れ→160gt 後に再点灯試行、という
  状態を追加する (02 §6 torch のロジックをそのまま)。純ロジックで更新順・エンティティ非依存。
- **仕様確定状況: [確定]**。02 §6 torch: 定数 `RECENT_TOGGLE_TIMER=60` / `MAX_RECENT_TOGGLES=8` /
  `RESTART_DELAY=160`、消灯のたびに記録追加・8 件で焼き切れ・160gt 後再点灯 (その時も 8 件なら再点灯せず)。
  デコンパイル済みで追加調査不要。
- **推奨: 実装**。仕様が確定しており実機一致に寄与。**割当案 = 新規 I12「STC タイミング精緻化」** (C9 ランプ遅延と同居)。
- **ユーザ判断ポイント**:
  1. 焼き切れを再現するか (再現すると、ユーザが速いトーチクロックを組んだとき「勝手に止まる」挙動になる。
     教育的には正しいが「動かない」と誤解される UX リスクあり → 表示/トースト等の補助を検討)。
  2. トグル履歴の保持コスト (トーチ毎の時刻リスト) を許容するか (回路規模では軽微)。

## 8. C8: ダスト dot 形状

- **回路勢重要度: 低〜中**。ダストの**接続形状が給電方向を決める**ため、論理に効くのは主に
  「直線 (例 N-S 接続) は E/W 隣接を給電しない」ルール (02 §5.4 [確定]: 給電対象 = **接続方向の水平隣接のみ**)。
  これは既に「接続」を明示モデル化していれば給電ロジック側で担保される。純粋な **dot (全方向 none)** は
  「結線先が何も無い孤立ダスト」で、そもそも給電相手が居ないため論理的差分は小さい。残る差分は
  **見た目 (cross/dot テクスチャ)** と、隣に素子が現れたとき dot が自動で結線し直す**形状の自動計算**。
- **実装コスト: 小 (主に描画 + 形状計算)**。本 sim は接続を「配置時に確定・実行中不変」でモデル化 (types.ts
  WireState.connections)。給電方向は接続から導出済み。よって残るのは (a) viewer での dot/cross 描画、
  (b) 近傍の素子/ダストに応じた形状の**自動計算**を **PP/形状更新 (I6 #14)** に載せるか。tile tick 非依存。
- **仕様確定状況**: 接続→給電方向 = 02 §5.4 [確定]。形状 (dot/side/up) は PP で維持 = 02 §6 wire [確定]。
  wiki の dot 記述は取得できず [要検証] だが、論理は 02 で足りる。
- **推奨: 実装 (低優先)**。ロジック影響は接続モデルで概ね吸収済み。**割当案 = I6 (#14) 相乗り**
  (形状の自動計算・PP 維持) + viewer 側の描画。単独 issue 化までは不要。
- **ユーザ判断ポイント**:
  1. 接続を「配置時確定・不変」の現設計のまま、dot/line/cross の**自動再計算**を I6 でどこまで持つか
     (実行中に隣接が変わらない前提なら描画のみでも実害小)。
  2. dot の描画差 (視認性) を viewer 課題として別で扱うか。

## 9. C9: ランプ消灯 4gt 遅延

- **回路勢重要度: 低 (下流影響なし) / 中 (実機 fixture 精度)**。ランプは**電源ではない** (信号を出力しない) ため
  消灯タイミングが下流回路に波及しない。しかし I4 の実機 fixture 生成中に**未実装ギャップとして発見**され
  (PR#28)、ランプを含む状態系列 diff が tick22 以降で実機とズレる原因になった。**fixture の tick 単位一致**の
  ためには要実装。
- **実装コスト: 極小**。依存は **I3 (#11, tile tick) 済**。点灯は NC 即時のまま、消灯を「4gt の tile tick を予約し
  tick 時に入力再評価してまだ無入力なら消灯」に変える (torch/repeater と同型の再評価パターン)。1 ブロック分の
  小改修。
- **仕様確定状況: [確定]**。02 §6 lamp: 「点灯=NC 即時。消灯=4gt tile tick 予約 + tick 時再評価、4gt 未満の
  入力断では消灯しない」。デコンパイル済み (未解明 #4 解消)。
- **推奨: 実装**。仕様確定・低コスト・fixture 一致に直結。**割当案 = 新規 I12「STC タイミング精緻化」**
  (C7 トーチ burnout と同居させると、いずれも I3 後の「tile tick 再評価型タイミング」精緻化で性質が揃う)。
- **ユーザ判断ポイント**:
  1. C7 と束ねて I12 にするか、ランプ単体の最小 issue にするか (どちらも極小)。
  2. ランプは出力を持たないため優先度は低め — I5/I6 の合間の埋め草として扱ってよいか。

## 10. スコープ外宣言文 (README 追記草案)

スコープ外にするコンポーネント (C2 / C3 自動 / C4 / C5 / C6' 物流) を README「対応ブロック」節の後に
以下の一段落として宣言する案。実装寄り (C1/C6/C7/C8/C9) は各 issue 完了時に「対応ブロック」へ加える。

> ### スコープ外のコンポーネント
>
> redstone-sim は**静的なレッドストーン論理**のシミュレータであり、エンティティ・投射物・アイテム物流・
> 音声を伴う挙動は対象外です。具体的に、以下は (少なくとも現段階では) 実装しません:
>
> - **感圧板 (木・石・重量式)** および **ターゲットブロック** — 発火にエンティティや投射物の物理が必要で、
>   本シミュレータはこれらを持ちません。入力が要る場合はレバー / ボタンで代替してください。
>   (重量式のアナログ出力はコンパレーター + コンテナ充填で代替できます。)
> - **音符ブロック** — レッドストーン信号を出力せず、機能は音声再生のため、回路論理には影響しません。
> - **コンテナのアイテム物流** (ホッパーのアイテム転送・ドロッパー/ディスペンサーの発射・アイテム移動) —
>   アイテム/インベントリのシミュレーションは対象外です。**コンパレーターによる充填率の読み取り**は
>   対応予定で、その際コンテナの中身は「充填率を直接指定する」形で扱います。
>
> ピストン・オブザーバー・準接続 (QC) は**スコープ内**で、実装ロードマップ (docs/research/04 §3 の I7・I8) に
> 含まれます。判断の根拠は docs/research/10_component-scope.md を参照。
